# PRD: セットマージャン成績管理システム「雀録 (JanRoku)」

## 1. プロダクト概要

### 1.1 プロダクト名
**雀録 (JanRoku)** — セットマージャン成績記録・分析システム

### 1.2 ビジョン
仲間内のセットマージャンにおいて、対局中のリアルタイムスコア記録から清算、長期的な成績蓄積・分析・レーティングまでをワンストップで提供するモバイルファーストのWebアプリケーション。

### 1.3 対象ユーザー
- セットマージャン（友人・知人間での定期的な麻雀会）を行うグループ
- 主に4人打ち半荘戦を中心にプレイするメンバー
- スマートフォンで対局中にスコアを入力する操作者（記録係）

### 1.4 解決する課題
- 紙の記録は紛失・集計ミスが発生しやすい
- 対局中のスコア入力が煩雑で時間がかかる
- 長期的な成績データが残らず、振り返りや分析ができない
- 清算計算を手動で行うとミスが起きやすい
- メンバー間の実力差を客観的に把握できない

## 2. 技術要件

### 2.1 システムアーキテクチャ
- **フロントエンド**: Next.js (App Router) + TypeScript + Tailwind CSS
- **バックエンド**: Next.js API Routes
- **データベース**: SQLite (better-sqlite3) — ローカルファイルベースで堅牢
- **ORM**: Drizzle ORM
- **ホスティング**: ローカル起動 or VPS（Dockerコンテナ対応）

### 2.2 データ永続化戦略（最重要要件）
対局中のデータ喪失は絶対に防がなければならない。以下の多層防御を採用する。

1. **サーバーサイドSQLite**: メインのデータストア。WALモード有効化で読み書き並行性を確保
2. **トランザクション保証**: すべてのスコア書き込みはトランザクション内で実行
3. **楽観的UI + 即時永続化**: UIは即座に更新しつつ、バックグラウンドでAPIを叩いてDB保存。保存完了まで「保存中」インジケーター表示
4. **ローカルストレージバックアップ**: ネットワーク障害時にlocalStorageに一時キャッシュし、復旧時に自動同期
5. **操作ログ（WAL的設計）**: 各スコア操作をイベントログとして記録し、万が一の復元に対応
6. **自動バックアップ**: SQLiteファイルの定期自動バックアップ（日次）

### 2.3 モバイルファースト設計方針
- 全画面タッチ操作最適化（最小タップターゲット: 44x44px）
- 片手操作を考慮したUI配置
- スコア入力は最小タップ数で完了する設計
- PWA対応（ホーム画面追加、オフライン基本動作）
- 横向き・縦向き両対応

## 3. 機能要件

### 3.1 コア機能① — 対局セッション管理（リアルタイム帳表）

#### 3.1.1 セッション作成
- 日付（デフォルト: 当日）の設定
- 参加メンバーの選択（登録済みメンバーリストから選択 / 新規追加）
- ルール設定
  - 持ち点（デフォルト: 25,000点）
  - 返し点（デフォルト: 30,000点）
  - ウマ設定（デフォルト: 10-30 / カスタム可能）
  - 順位点（オカ）の自動計算
  - チップ（祝儀）の有無と単価
  - レート設定（点ピン、テンゴ等 / 表示上の参考値として）
  - 赤ドラの有無

#### 3.1.2 半荘記録
- **素点入力画面**: 4人の素点を入力（合計チェック機能付き）
  - テンキーUIで高速入力
  - 合計が10万点（4人分）にならない場合は警告表示
  - 前の半荘の結果からの差分表示
- **自動計算**:
  - 順位の自動判定
  - ウマの自動付与
  - オカ（返し点との差分）の自動計算
  - ポイント換算の自動計算
- **チップ記録**: 各半荘ごとのチップ枚数を記録（任意）
- **トビ（箱割れ）判定**: 0点以下でのトビを検出
- **半荘の修正・削除**: 入力ミスへの対応（修正履歴を保持）

#### 3.1.3 リアルタイム帳表表示
- **スコアボード**: 現在のセッション内の全半荘を一覧表示
  - 各半荘の素点・順位・獲得ポイント
  - セッション通算のトータルポイント
  - チップ合計
- **ランキング表示**: セッション内での現在の順位
- **収支サマリー**: リアルタイムの収支状況（レート換算含む）

#### 3.1.4 清算機能
- セッション終了時の最終精算
  - 各メンバーのトータルポイント集計
  - チップの集計（ある場合）
  - レート換算による金額計算
  - **清算表の生成**: 「AはBにX円支払い」形式の最適化された送金指示
    - 送金回数を最小化するアルゴリズム（ネッティング処理）
- 清算完了の確認・ロック機能
- 清算結果の共有（テキストコピー / LINEシェア用テキスト生成）

#### 3.1.5 対局中の補助機能
- 起家マーク表示
- 半荘番号の自動カウント
- 対局時間の記録（セッション開始〜終了）
- メモ機能（対局中の出来事を記録）

### 3.2 コア機能② — データ蓄積・分析・レーティング

#### 3.2.1 成績データベース
- 全セッションの清算済みデータを永続保存
- メンバーごとの全成績を時系列で管理
- データのエクスポート機能（CSV / JSON）

#### 3.2.2 個人成績ダッシュボード
- **基本統計**:
  - 通算対局数（半荘数）
  - 通算収支（ポイント / 金額換算）
  - 平均順位
  - 1着率 / 2着率 / 3着率 / 4着率（順位分布）
  - トビ率
  - 平均獲得ポイント
  - 最高獲得ポイント / 最低獲得ポイント
  - 連対率（1着+2着率）
- **グラフ表示**:
  - 収支推移グラフ（折れ線グラフ）
  - 順位分布（円グラフ / 棒グラフ）
  - 月別成績推移
- **期間フィルター**: 全期間 / 年別 / 月別 / 直近N回 でのフィルタリング

#### 3.2.3 レーティングシステム
以下のハイブリッドレーティングを実装する。

**ベースレーティング（Glicko-2ベース）**:
- 初期レート: 1500
- 各半荘の順位結果に基づいてレーティングを更新
- Rating Deviation (RD) を導入し、対局数が少ないプレイヤーの変動幅を大きくする
- Rating Volatility (σ) により急激な実力変動にも対応
- 長期間対局がないメンバーはRDが徐々に増加（不確実性の増大）

**レーティング計算ロジック**:
- 4人対戦を6つのペアワイズ比較（1v2, 1v3, 1v4, 2v3, 2v4, 3v4）に分解
- 各比較で「勝ち=1.0 / 負け=0.0」として期待勝率との差分でレート更新
- 順位だけでなく素点差も考慮する重み付けオプション

**レーティング表示**:
- 現在のレーティング値
- レーティング推移グラフ
- レーティングランキング（全メンバー）
- 信頼区間の表示（RDに基づく±範囲）
- レーティングの変動履歴

#### 3.2.4 グループ分析
- **対戦相性マトリクス**: メンバー間の対戦成績を表示
- **セッション履歴**: 過去の全セッション一覧と詳細
- **出席統計**: メンバーごとの参加頻度
- **グループランキング**: 各種指標でのランキング（通算収支、平均順位、レーティング等）
- **月間MVP**: 月間最優秀成績者のハイライト

### 3.3 追加機能（あったらいい機能）

#### 3.3.1 メンバー管理
- メンバープロフィール（名前、アバター/イニシャル、メモ）
- メンバーの追加・編集・非アクティブ化（削除はせず論理削除）
- ゲスト参加機能（一時メンバー）

#### 3.3.2 ルールプリセット
- よく使うルール設定をプリセットとして保存
- プリセットの切り替えでセッション作成を高速化
- 複数グループに対応（異なる仲間内ごとのプリセット）

#### 3.3.3 通知・リマインド
- セッション未清算アラート
- データ入力忘れ防止リマインド

#### 3.3.4 役満・特殊イベント記録
- 役満の記録（誰が / いつ / 何の役満）
- 特筆すべきイベントの記録機能
- 役満達成者の殿堂表示

#### 3.3.5 データ共有
- セッション結果をテキストベースで共有（コピー&ペースト向け）
- 成績サマリーの画像生成（SNS共有用）

#### 3.3.6 ダークモード
- 対局環境（薄暗い部屋等）に配慮したダークモード対応
- デフォルトはダークモード推奨

#### 3.3.7 五人打ち対応
- 5人で集まった場合の抜け番ローテーション管理
- 抜け番を考慮した公平なスコアリング

## 4. 画面構成

### 4.1 画面一覧

| 画面名 | パス | 概要 |

### 4.2 UI/UXの方針
- **カラーテーマ**: 麻雀卓の緑を基調とした落ち着いたダークテーマ
- **フォント**: 数字の視認性を最重視（等幅数字フォント）
- **アニメーション**: スコア更新時の軽微なアニメーションで視認性向上
- **ナビゲーション**: 下部タブバー（ホーム / 対局 / 履歴 / 分析 / 設定）

## 6. レーティングシステム詳細

### 6.1 Glicko-2 パラメータ
- **初期レーティング (μ)**: 1500
- **初期RD (φ)**: 350（新規プレイヤーは変動が大きい）
- **初期Volatility (σ)**: 0.06
- **システム定数 (τ)**: 0.5（変動の激しさの制御）
- **RD上限**: 350（長期不在時の上限）
- **RD下限**: 50（安定プレイヤーの下限）

### 6.2 更新タイミング
- 各半荘終了時にリアルタイム更新
- セッション清算時に最終確定

### 6.3 表示ランク（任意のラベル）
| レーティング範囲 | ランク名 |
|-----------------|---------|
| 1800+ | 雀聖 |
| 1700-1799 | 雀豪 |
| 1600-1699 | 雀傑 |
| 1500-1599 | 雀士 |
| 1400-1499 | 初段 |
| 1300-1399 | 1級 |
| 1200-1299 | 2級 |
| ~1199 | 3級 |

## 7. 非機能要件

### 7.1 パフォーマンス
- スコア入力から保存完了まで500ms以内
- 帳表の再描画は100ms以内
- 成績ダッシュボードの読み込みは2秒以内

### 7.2 データ安全性
- SQLite WALモードによるクラッシュリカバリ
- 全操作のイベントログ記録
- 日次自動バックアップ
- localStorage による一時キャッシュ（ネットワーク障害対策）

### 7.3 対応環境
- iOS Safari 15+
- Android Chrome 90+
- デスクトップブラウザ（Chrome, Safari, Firefox 最新版）

### 7.4 セキュリティ
- 本システムは閉じたプライベート環境での使用を前提とする
- 将来的に認証機能を追加する場合に備え、APIルートを分離設計
