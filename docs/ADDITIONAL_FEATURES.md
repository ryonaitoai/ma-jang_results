## 追加機能1: リアルタイム閲覧モード（観戦URL共有）

### 背景・ユースケース
セットマージャンでは1人が記録係としてスコアを入力するが、他の3人は「今どのくらいのスコアなんだろう」と気になる場面が頻繁にある。また、入力されたスコアを他のメンバーが確認することで、入力ミスの早期発見にもつながる。

### 要件

#### 閲覧用URL
- 進行中のセッションには、共有用の閲覧URLを発行する
  - 形式: `/sessions/[id]/view` または `/s/[短縮ID]`
  - このURLはスマートフォンのブラウザで開くだけで、ログインや認証なしに閲覧できる
- 記録係のセッション画面に「共有」ボタンを設置し、URLのコピーまたはQRコードを表示する
  - QRコードは対局前に他の3人がサッと読み取れるようにする

#### 閲覧画面の内容
- 現在のセッションのスコアボード（帳表）をリアルタイムで表示
  - 各半荘の結果（入力済みスコア、順位）
  - トータルポイントの累計
  - チップ合計（有効な場合）
  - 現在の順位（ハイライト表示）
- **読み取り専用**: 閲覧者はデータの編集・入力はできない
- 最新データの反映は **ポーリング方式** で実装する
  - 5秒間隔で `/api/sessions/[id]/scores` をGETリクエスト
  - レスポンスに変更がない場合はUIを更新しない（差分比較）
  - 将来的にServer-Sent Events (SSE) への移行を考慮したAPI設計にしておく
- 新しい半荘が追加されたときに、軽いアニメーションで通知する（画面上部にトースト等）

#### 技術的な考慮
- ポーリングはサーバー負荷が低いSQLiteリードで十分に軽い
- APIレスポンスには `lastUpdated` タイムスタンプを含め、クライアント側で変更検知に使用する
- 閲覧画面はPWAのフルスクリーンモードでも快適に動作すること
- 対局卓で全員のスマホが同時にアクセスするため、同時接続4-5台で安定動作すること

#### 画面設計
- 閲覧画面は記録係の帳表画面と見た目をほぼ同一にする（一貫性のため）
- ただし入力ボタン・編集機能は非表示
- 画面上部に「LIVE 🔴」のようなリアルタイム閲覧中インジケーターを表示
- 接続状態の表示（最終更新: X秒前）

## 追加機能2: ポイント入力方式への変更（実運用フロー最適化）

### 背景・ユースケース
実際のセットマージャンでは、半荘終了時に **2着・3着・4着の3人がウマ・オカ込みのポイントを口頭で自己申告** し、記録係がそれを記録する。**トップ（1着）の数字は申告しない** — 合計が0になるよう逆算すれば自動的に決まるためである。

つまり実際の流れは:
1. 半荘終了
2. 2着「プラス8」、3着「マイナス23」、4着「マイナス50」と申告
3. 記録係が3人分を入力
4. トップは `0 - (8 + (-23) + (-50)) = +65` と自動計算される

この **「3人入力 → トップ自動計算」** という実運用フローをそのままUIに反映し、最小タップ数での入力を実現する。

### 要件

#### 入力方式の変更
- **メイン入力**: ウマ・オカ込みの最終ポイントを直接入力する
  - 例: `+8`, `-23`, `-50` → トップ `+65` は自動計算
  - 入力する数字は小数点以下も対応（例: `+3.5`, `-12.5`）
- **素点入力は補助モードとして残す**: 切替トグルで「素点入力モード」にも切り替えられるようにする（従来のPRDの方式）
- デフォルトはポイント入力モード

#### 入力画面の構成（最重要）

画面レイアウト（上から下）:

```
┌─────────────────────────────────┐
│  半荘 #3  スコア入力             │
├─────────────────────────────────┤
│                                 │
│  🥇 太郎（トップ）               │
│  ┌─────────────────────────┐   │
│  │    +65.0    ← 自動計算   │   │
│  └─────────────────────────┘   │
│  ※ 他3人の入力で自動算出        │
│                                 │
│  2着 次郎        [  +8  ]  ← 入力欄 │
│  3着 三郎        [ -23  ]  ← 入力欄 │
│  4着 花子        [ -50  ]  ← 入力欄 │
│                                 │
│  ─── 合計チェック: 0.0 ✓ ───    │
│                                 │
│  ┌─────────────────────────┐   │
│  │        確定              │   │
│  └─────────────────────────┘   │
│                                 │
│  ┌───┬───┬───┐               │
│  │ 1 │ 2 │ 3 │               │
│  ├───┼───┼───┤               │
│  │ 4 │ 5 │ 6 │               │
│  ├───┼───┼───┤               │
│  │ 7 │ 8 │ 9 │               │
│  ├───┼───┼───┤               │
│  │ . │ 0 │ ⌫ │               │
│  ├───┴───┴───┤               │
│  │    ± 符号切替    │               │
│  └───────────┘               │
└─────────────────────────────────┘
```

#### ステップ1: トップ（1着）の指定

- 入力画面を開くと、まず **「トップは誰？」** の選択UIを表示する
  - 4人のメンバー名がボタンとして横並び（または縦2×2グリッド）
  - 1タップでトップを選択 → 入力画面に遷移
- トップに選ばれたメンバーは **自動計算欄** として表示される（入力不可）
- 残りの3人が入力対象として並ぶ

**なぜトップを先に聞くか**: トップの人は申告しないので、「誰がトップか」だけは記録係が把握している必要がある。実運用ではトップが誰かは全員わかっているので、1タップで済む。

#### ステップ2: 2着〜4着のポイント入力

- トップ以外の3人の入力欄が縦に並ぶ
- **順位は入力不要**: ポイントの大小で順位は自動判定できる（ポイントが大きい方が上位）
- 各入力欄のUX:
  1. メンバー名の行をタップ → その行がアクティブ（ハイライト）になりテンキーでフォーカス
  2. テンキーで数字を入力（例: `23`）
  3. **デフォルトはマイナス**: 2-4着はマイナスであることが多いので、初期状態を **マイナス** にする
  4. プラスの場合は `±` ボタンをタップして反転 → `-23` が `+23` に変化
  5. 入力確定は次のメンバー行をタップするか、テンキー外をタップ

**デフォルトマイナスの理由**: 2着〜4着は統計的にマイナスになることが多い（特に3着・4着）。2着でプラスの場合だけ `±` をタップする方が、毎回マイナスを手動で切り替えるよりタップ数が少ない。

#### ステップ3: トップの自動計算とバリデーション

- 3人分のポイント入力が進むにつれ、トップの欄にリアルタイムで自動計算値が表示される
  - 計算式: `トップのポイント = 0 - (2着 + 3着 + 4着)`
  - 1人目入力中から逐次計算（未入力は0扱い）
- トップの自動計算値が妥当かどうかの目安表示:
  - 値が正の場合: 緑色で表示（正常）
  - 値が負の場合: オレンジ色で警告（トップがマイナスは稀なので注意喚起。ただしありえなくはないのでブロックはしない）
  - 極端に大きい値（例: ±100以上）: 黄色で「要確認」表示

#### 符号トグルのUX

マイナス符号の入力は煩雑なので、以下のトグル方式を採用する:

1. **初期状態はマイナス**（2着〜4着の入力欄）: 数字入力欄は最初 `-` 表示
2. **数字を入力**: テンキーで数字を入力（例: `23`）→ 表示は `-23`
3. **符号トグル**: 以下のいずれかの操作で符号が反転する
   - `±` ボタンをタップ
   - 入力済みの数字表示エリアをタップ
   - → `-23` ↔ `+23` とトグル
4. **視覚的フィードバック**:
   - プラス時: 数字を **緑色** で表示、背景も薄い緑
   - マイナス時: 数字を **赤色** で表示、背景も薄い赤
   - 符号トグル時に色が切り替わるアニメーション
5. **トップの自動計算欄**: 入力不可だが、符号に応じて同様の色分けを適用

#### テンキーUI

```
[ 1 ] [ 2 ] [ 3 ]
[ 4 ] [ 5 ] [ 6 ]
[ 7 ] [ 8 ] [ 9 ]
[ . ] [ 0 ] [ ⌫ ]
[    ± 符号切替     ]
```

- `±` ボタンは大きく、親指で押しやすい位置に配置（テンキーの最下部、フル幅）
- テンキーは画面下部に固定表示（入力中はスクロールしない）
- `.`（小数点）ボタンは小数点以下のポイント入力用
- `⌫` で1文字ずつ削除、長押しで全消去

#### 全体の入力フロー（まとめ）

```
「+半荘追加」タップ
    │
    ▼
「トップは誰？」— 4人のボタンから1タップで選択
    │
    ▼
入力画面表示:
  - トップ: 自動計算欄（グレーアウト）
  - 残り3人: 入力欄（デフォルト: マイナス）
    │
    ▼
1人目の行をタップ → テンキーフォーカス → 数字入力 → (必要なら±トグル)
    │
    ▼
2人目の行をタップ → 同上（1人目は自動確定）
    │
    ▼
3人目の行をタップ → 同上
    │
    ▼
トップの値が自動計算で表示される
合計チェック: 0.0 ✓
    │
    ▼
「確定」ボタンタップ → DB保存 → 帳表に戻る
```

**最小タップ数の分析**:
- トップ選択: 1タップ
- 3人分の数字入力: 各2-3タップ（数字キー） × 3人 = 6-9タップ
- 符号切替（必要な場合のみ）: 0-3タップ
- 確定: 1タップ
- **合計: 最短8タップ、典型的に12タップ程度で1半荘の記録が完了**

#### 順位の自動判定

- ポイント入力モードでは順位を手動入力しない
- 全員のポイントが確定した時点で、ポイントの大きい順に1着〜4着を自動判定
- 同点の場合は席順（session_membersのseat_order）で上座を上位とする
- 判定結果は確定前に表示し、記録係が目視で確認できるようにする

#### 合計チェック

- **ポイント入力モードでは合計は構造上必ず0になる**（トップが自動計算のため）
- ただし、トップの値を記録係が手動で上書き修正できるオプションも設ける（レアケース対応）
  - トップ欄をタップすると「手動入力に切替」のオプションが出る
  - 手動入力に切り替えた場合のみ、合計0チェックを実施
  - 合計が0でない場合は警告ダイアログ表示（保存はブロックしない）

#### データモデルへの影響
- `hanchan_scores` テーブル:
  - `raw_score` (素点): NULLを許容する（ポイント入力モードでは素点がないため）
  - `point` (ポイント): 新規カラム追加。直接入力されたポイント値を保存
  - `input_mode`: `'point'` | `'raw_score'` — どちらのモードで入力されたかを記録
  - `is_auto_calculated`: BOOLEAN — トップの自動計算フラグ（ポイント入力モードの1着に対してtrue）
- 素点入力モードの場合は従来通り `raw_score` から `point` を自動計算
- ポイント入力モードの場合は `point` を直接保存し、`raw_score` はNULL
- `hanchan` テーブルに `top_member_id` (FK) を追加 — トップとして選択されたメンバーを記録

## PRD・プロンプトへの反映指示

1. **セッション画面 (`/sessions/[id]/page.tsx`)** に「共有」ボタン（URL コピー + QRコード表示モーダル）を追加
2. **新規ルート `/sessions/[id]/view`** を読み取り専用の閲覧画面として作成
3. **半荘入力画面 (`/sessions/[id]/input/page.tsx`)** を「トップ選択 → 3人入力 → トップ自動計算」のフローに全面改修
4. **APIエンドポイント** に `GET /api/sessions/[id]/scores` を追加（ポーリング用、`lastUpdated` 付き）
5. **DBスキーマ**:
   - `hanchan_scores` テーブルに `point`, `input_mode`, `is_auto_calculated` カラムを追加、`raw_score` をNullable化
   - `hanchan` テーブルに `top_member_id` カラムを追加
6. **スコア計算ロジック (`calculator.ts`)** にポイント入力モード時のトップ自動計算と順位自動判定ロジックを追加
